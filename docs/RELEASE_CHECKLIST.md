# Release Checklist

## 当前结论

截至 `2026-03-15`，这个项目的状态是：

- `可以`用于内部演示、汇报、方案说明、小范围试用
- `不建议`直接作为正式公网产品上线

原因不是界面不够，而是当前还存在几个明确的发布阻塞项。

## 当前阻塞项

| 项目 | 当前状态 | 是否阻塞正式上线 | 说明 |
| --- | --- | --- | --- |
| 数据真实性 | 未锁定真实数据模式 | 是 | 前端和后端都存在 `API/数据库失败 -> 自动回退 mock 数据` 的逻辑。如果公网环境里仍保留这个行为，用户可能把演示数据当成真实数据。 |
| 服务部署方式 | 仍是本地开发形态 | 是 | 当前 PowerShell 服务直接监听 `http://localhost:<port>/`，适合本地运行，不适合作为公网发布形态。 |
| 发布验证 | 未发现自动化测试或正式验收记录 | 是 | 当前仓库中未发现测试配置或 E2E 验收脚本，说明上线前至少还缺一轮系统验证。 |
| 异常可见性 | 失败时降级优先，错误暴露不足 | 是 | 现在的主要策略是“尽量显示内容”，对正式产品来说应改成“真实数据不可用时明确报错”。 |
| 监控与运维 | 未见生产级监控/告警/日志方案 | 高风险 | 本地有 `server-out.log` / `server-err.log`，但未形成正式的线上日志与告警闭环。 |
| 对外说明 | 缺少显式的数据说明与免责声明 | 建议补齐 | 如果对外发布，建议明确说明数据来源、更新时间、分数含义、适用范围仅限美国。 |

## 仓库中已确认的事实

### 1. 前端存在 mock 回退

- 文件：`api-client.js`
- 行为：请求 `/api/airs/*` 失败后，自动回退到 `mock-data-v2.js`

这意味着只要后端异常，页面仍能继续显示内容，但内容不一定是真实数据。

### 2. 后端存在 mock 回退提示

- 文件：`backend/server.ps1`
- 当前行为：当数据库模式未配置时，控制台会输出：
  - `AIRS database mode is not configured. API routes will use mock data.`

这说明当前服务本身就支持“数据库未配置时继续用 mock 数据”。

### 3. 当前服务监听的是本地地址

- 文件：`backend/server.ps1`
- 当前行为：监听 `http://localhost:$Port/`

这属于本地开发服务，不是正式公网服务形态。

### 4. 当前仓库中未发现自动化测试配置

已检查但未发现下列常见配置文件：

- `package.json`
- `playwright.config.*`
- `*.spec.*`
- `*.test.*`
- `pytest.ini`
- `pyproject.toml`

这不代表不能上线，但代表目前没有看到最基本的自动化发布验收支撑。

## 正式上线前必须完成

### 1. 锁定真实数据模式

要求：

- 生产环境禁止静默回退到 mock 数据
- 真实数据不可用时，页面要明确提示“数据暂时不可用”
- 首页和详情页都必须能确认当前模式是 `实时数据`，不是演示数据

建议动作：

1. 增加 `production` 开关
2. 在生产模式下关闭前端 mock fallback
3. 在后端数据库不可用时返回明确错误码和错误信息

当前仓库进度：

- `已具备基础能力`
- 现已支持严格数据模式：
  - 前端可通过 `airs-strict-data-mode` 开关禁用 mock fallback
  - 后端可通过 `.\start.ps1 -StrictDataMode` 返回 `503`
  - 页面已支持明确展示“实时数据不可用”

仍需你在正式环境里真正启用，而不是只保留默认配置。

### 2. 改成正式部署形态

要求：

- 使用正式域名
- 全站 HTTPS
- 通过 IIS / Nginx / 反向代理发布
- 端口、日志、异常页都有明确配置

建议动作：

1. 反向代理到 PowerShell API
2. 只暴露公网入口，不直接暴露本地监听端口
3. 配置 gzip / cache / 静态资源缓存头

### 3. 完成最基本的验收测试

至少覆盖以下路径：

1. 首页打开正常
2. 中文/英文切换正常
3. 日期、地区、职业大类、影响类型筛选正常
4. 搜索正常
5. 点击职业节点后详情页跳转正常
6. 详情页趋势图、证据、任务暴露正常
7. 接口失败时页面能明确报错，不显示伪实时内容
8. 桌面端与手机端样式不破版

### 4. 完成数据解释与对外说明

建议在页面底部或说明页补充：

- 数据覆盖范围：`美国`
- 职业分类标准：`BLS SOC`
- AIRS 分数解释：
  - `100` = AI 当前影响较小
  - `0` = AI 已高度替代该职业相关人力需求
- 更新频率
- 数据源名称
- 免责声明：结果用于观察招聘变化，不等于官方就业预测

当前仓库进度：

- `页面层说明区已具备`
- 首页和详情页已加入：
  - 数据模式
  - 运行环境
  - 最近更新时间
  - 覆盖范围
  - 分数含义
  - 免责声明

仍需你确认最终对外文案和真实数据源名称。

## 建议上线前完成

这些不是第一天就必须有，但建议在正式发布前补齐：

### 1. 监控

- 服务存活监控
- API 错误率监控
- 数据更新失败告警
- 日志轮转

### 2. 性能

- 首页首屏加载时间
- 接口缓存
- 静态资源压缩
- 长列表/大量节点下的交互流畅度

### 3. 安全

- HTTPS 强制跳转
- 基本安全响应头
- 反向代理访问限制
- 日志中不输出密钥

## 当前最小可上线路径

如果你的目标是“尽快从演示版走到可正式开放访问”，建议按下面顺序推进：

1. 接好真实数据源，并确认页面不再使用 mock fallback
2. 把 PowerShell 服务放到正式反向代理后面，启用 HTTPS
3. 做一轮人工验收
4. 补上数据说明页或页脚说明
5. 再决定是否对公网开放

## 当前可发布级别判断

### 可以立即做的

- 内部演示
- 向领导/客户展示产品方向
- 小范围试用
- 用于继续接入真实数据源

### 不建议现在做的

- 直接公网开放
- 对外宣称“实时正式版”
- 在未禁用 mock 回退前作为正式商业产品发布

## 上线前最后一问

只有下面 4 个问题都能回答“是”，才建议正式上线：

1. 当前页面显示的数据，是否可以确认全部来自真实数据源？
2. 数据源失败时，页面是否会明确报错而不是悄悄显示 mock 数据？
3. 是否已经通过桌面端和手机端的人工验收？
4. 是否已经具备正式域名、HTTPS 和反向代理部署？

如果其中任意一项回答是否定的，建议继续以内测/演示状态运行。
